## 1. 项目全景（先建立心智模型）

### 1.1 一句话定位

PseudoFS 在**内存里模拟一块磁盘**（一段 `calloc` 出来的字节数组），在它上面实现**两套**真实文件系统的核心算法（UNIX inode 体系、FAT16），用一个 **VFS 抽象接口**把它们统一起来，再套一层 **Shell 命令解析**和 **ncurses TUI**。它**从不碰宿主机真实文件**，唯一的真实 I/O 是把整块「磁盘」dump 成一个 `.img` 文件做持久化。

### 1.2 五层架构（自顶向下）

```

│ TUI 层   src/tui/tui.cpp            ncurses 四面板 + 功能键 │  ← B
├─────────────────────────────────────────────────────────┤
│ Shell 层 src/shell/command_registry 命令注册/分词/glob/分发 │  ← B
```

### 1.3 一条命令的完整生命（端到端追踪）

这是理解全局最快的方式。追 `write 3 hello`（往 fd=3 写 "hello"）：

1. **TUI/CLI** 读到一行字符串 `"write 3 hello"`
   - CLI：`main.cpp` 主循环 `std::getline` → `reg.execute(line, *fs, um, output)`
   - TUI：`tui.cpp` 的 `wgetch` 事件循环攒出整行，回车时同样调 `reg_->execute(...)`
2. **Shell** `CommandRegistry::execute`（`command_registry.cpp:56`）：
   - `tokenize_ex` 分词（处理引号/转义）→ `["write","3","hello"]`
   - `expand_globs` 对最后一段做通配符展开（这里无 `*`，原样）
   - 取首 token `write` 查命令表 → 找到对应 `CmdHandler` lambda
   - `record()` 记操作日志（用于 `replay`）
   - 调 handler
3. **命令 handler**（`main.cpp` 里注册的 write 处理，约 `:607`）：把 `3` 转成 fd、把后面拼成数据，调 `fs.fs_write(3, "hello", 5)`
4. **VFS → UNIX 引擎** `UnixFs::fs_write`（`unix_fs.cpp:228`）：
   - 查打开文件表拿到 inode 号和当前偏移 `offset`
   - `imng_.get(ino)` 把磁盘 inode 读进内存（`MemINode`）
   - 循环：对每个要写的逻辑块，`bmap_alloc(ip, blk_idx)` 把「文件第 N 块」映射到「数据区物理块号」，**没有就分配**（可能触发间址块分配 → `sb_.balloc()` 成组链取块）
   - `dev_.read_block` / `memcpy` / `dev_.write_block` 落到「磁盘」
   - 更新 `di_size`、`di_mtime`，`write_back` 写回 inode
   - `oft_.set_offset` 推进偏移，`sync()` 把整盘 dump 到 `.img`
5. **块设备** `BlockDevice::write_block`（`block_device.cpp:26`）：`memcpy` 到内存数组的 `block_no * 512` 偏移处。

> **读懂这条链 = 读懂整个项目。** 后面每一层都是在把这条链的某一段讲细。

### 1.4 关键数字速查（`include/core/constants.h`）

| 常量 | 值 | 含义 |
|------|----|----|
| `BLOCK_SIZE` | 512 | 块大小 |
| 磁盘布局 | boot[0] super[1] inode[2..33] data[34..545] | 共 `TOTAL_BLK_NUM=546` 块 ≈ 273KB |
| `INODE_SIZE` | 64 B | 每块 8 个 inode，共 `INODE_TOTAL=256` |
| `NADDR / DIRECT_ADDR` | 10 / 6 | addr[0..5] 直接，[6][7] 一次间址，[8] 二次间址，[9] 预留 |

| `MAX_FILENAME` | 14 | 目录项 16B = 14B 名 + 2B inode 号 |
| `MAX_OPEN_FILE/SYS_OPEN_FILE/MAX_USER` | 20 / 40 / 8 | 每用户/系统打开文件、最大用户数 |

**单文件最大长度** = 6×512 + 2×(256×512) + 1×(256×256×512) ≈ **3KB + 256KB + 32MB**（这就是必答题 Q4 的答案骨架）。

## 3. 块设备层 —— 地基（`block_device.{h,cpp}`）

**核心观念：所谓「磁盘」就是一段 `calloc` 出来的字节数组 `uint8_t* disk_`。**所有「磁盘操作」都是对这段内存按 `block_no * 512` 偏移做 `memcpy`。

```cpp
read_block(blk, buf):  memcpy(buf,  disk_ + blk*512, 512)
write_block(blk, buf): memcpy(disk_ + blk*512, buf,  512)
```
- 构造时 `calloc(total_blocks, 512)` —— **calloc 会清零**，所以新盘天然全 0。
- **持久化** = 把整段内存一次 `fwrite` 成 `.img` 文件（`save_to_file`），启动时 `fread` 整盘读回（`load_from_file`）。没有「写哪块存哪块」的增量持久化——简单粗暴但对课设足够。
- 禁拷贝（`= delete`），单线程，无块缓存（每次 read/write 直接动内存）。

> **为什么这样设计**：把「持久化」和「文件系统逻辑」彻底解耦。引擎只管 `read_block/write_block`，完全不知道数据最终落在内存还是文件。这正是真实 OS 里「块设备驱动」对文件系统隐藏物理细节的思想。

## 4. UNIX FS 引擎 —— 你的核心（重点章节）

四个文件，自底向上：`super_block`（空闲管理）→ `inode`（索引）→ `directory`（命名）→ `unix_fs`（把它们组装成 IFileSystem）。

### 4.1 磁盘布局（必画图 #1）
```
块号 0      1        2 ........... 33     34 ......................... 545
   ┌────┬────────┬───────────────────┬──────────────────────────────┐
   │boot│superblk│ inode 区 32 块     │ 数据区 512 块                  │
   │    │        │ 每块 8 个 inode    │ 成组块链接法管理               │
   │    │        │ 共 256 个 inode    │ 目录/文件数据/间址块都在这里    │
   └────┴────────┴───────────────────┴──────────────────────────────┘
```
- 物理块号 = `DATA_START_BLK(34) + 数据区相对块号`。引擎内部几乎都用**数据区相对块号**（0 起），落盘时才 `+34`。
- **关键不变量**：`fs_format` 里 `sb_.format(3, 2)` 预留了数据区前 3 块（相对 0/1/2）和 inode 0/1，于是 `balloc()` **永远不会发出相对块号 0**。所以 inode 里 `di_addr[i] == 0` 可以**无歧义地表示「这块没分配」**（空洞）。读到处理代码里 `if (phys == 0) 当作空洞` 全靠这个前提。

### 4.2 成组块链接法 —— 空闲块管理（OS 概念 #1，`super_block.cpp`）

**问题**：512 个数据块，哪些空闲？最朴素是位图，但课设 PPT 要求用 **UNIX 经典的成组链接法（grouped free list）**——既省空间又免遍历。

**数据结构**：超级块里有一个**只有 50 格的栈** `s_free[50]` + 计数 `s_nfree`。它只缓存「当前这一组」的空闲块号。其余成千上万的空闲块，分组**链在磁盘空闲块自己里**：

```
超级块内存栈(一组)          磁盘上的组(每组存在一个空闲块里)
┌──────────────┐
│ s_nfree = k  │
│ s_free[0] ───┼──link──▶ ┌────────────┐
│ s_free[1..k-1]│         │ count=50    │
│  = 空闲块号    │         │ entry[0]──link─▶ 下一组 ...
└──────────────┘          │ entry[1..49]│ = 空闲块号
                          └────────────┘
约定：s_free[0] / entry[0] 是「指向上一组的链」(0=链尾)；其余是真正的空闲块号。
```

**分配 `balloc()`（`super_block.cpp:99`）**：
- 栈里还有真块（`s_nfree > 1`）：直接弹栈顶 `s_free[--s_nfree]`。O(1)，不碰磁盘。
- 栈只剩链项（`s_nfree == 1`）：`s_free[0]` 是下一组所在的磁盘块号。**读那个块**把整组 50 个块号灌进内存栈，然后**把这个「装着组的块」本身作为空闲块返回**（它腾空了，正好拿去用）。
- `s_free[0] == 0`（链尾）且栈空 → 磁盘满，返回 `INVALID_BLK`。

**回收 `bfree(blk)`（`super_block.cpp:132`）**：
- 栈没满（`< 50`）：直接压栈。
- 栈满了（`== 50`）：**把当前满栈的 50 个块号写进「正要回收的这个块」里**，然后栈重置为只含这一个块（它成了新组的头）。——这就是「成组」的精髓：用被回收的块去存下一组的索引，不额外占空间。

**边界坑（值得在报告里写）**：`format` 建链时，如果空闲块数恰好是 50 的整数倍，最后会剩正好 50 块留给超级块内存栈，导致 `s_nfree = 50+1 = 51` 溢出 `s_free[50]`。所以代码故意「少建一组」，把 0..49 块留给内存栈、`s_nfree ≤ 50`。对应测试 `FormatFreeListIntactAtGroupBoundary`（`super_block.cpp:42-47` 注释）。

> **答辩高频题**：「成组链 vs 位图？」→ 成组链 O(1) 分配回收、空闲块索引存在空闲块自身里几乎零额外开销；缺点是**不支持快速判断某一具体块是否空闲**（要遍历整条链，见 `collect_free_blocks`），也不易做连续分配。位图正相反。

### 4.3 inode 与混合索引 —— 文件如何寻址（OS 概念 #2，`inode.cpp`）

**inode** 是「文件的元数据 + 数据块索引」，**不含文件名**（名字在目录项里）。磁盘 inode 64B（`DiskINode`，`inode.h:15`）：mode/nlink/uid/gid/size/三个时间戳 + `di_addr[10]` 地址数组。

**混合索引（mixed indexing）** 用 `di_addr[10]` 把「文件第 N 个逻辑块」映射到「数据区物理块号」。`bmap()`（`inode.cpp:174`）就是这个映射函数：

```
逻辑块号 N:
  N < 6                          → di_addr[N]                直接索引     (0~3KB)
  N < 6+256                      → di_addr[6] 指向的间址块[N-6]   一次间址 (块1)
  N < 6+512                      → di_addr[7] 指向的间址块[...]   一次间址 (块2)  (~256KB)
  N < 6+512+256*256              → di_addr[8] 二次间址:
        idx1=(N')/256 → 一级间址块[idx1] → 二级块[idx2=(N')%256] → 数据块   (~32MB)
  di_addr[9]                     预留(三次间址，未实现)
```
一个间址块 512B 装 `256` 个 `uint16_t` 块号（`PTRS_PER_BLK=256`）。

- `bmap`（只读）：没映射就返回 0（空洞）。
- `bmap_alloc`（`inode.cpp:212`，写时用）：路上缺哪一级（间址块或数据块）就 `balloc()` 补上，新间址块会**清零**（保证里面的指针都是 0=未分配）。这是 `fs_write` 能让文件「按需长大」的关键。
- `truncate`（`inode.cpp:299`）：删文件时**逐级释放**——先放直接块，再遍历两个一次间址块里的每个指针并释放、再释放间址块本身，最后处理二次间址（两层嵌套遍历）。漏放任何一级都会**块泄漏**。

> **答辩高频题**：「为什么要分级间址而不是全用直接？」→ 小文件（绝大多数）只用 `di_addr[0..5]`，零额外开销、一次寻址命中；大文件才付间址块的代价。这是「为常见情况优化」的经典权衡。「混合索引寻址过程」请照上面那张表口述一遍（必答题 Q1/Q4）。

### 4.4 inode 生命周期 —— iget / iput / 引用计数（OS 概念 #3，`inode.cpp`）

这是最容易讲错、也最能体现你懂 OS 的地方。两套计数别混：

- **`i_count`**（内存引用计数）：现在有多少个「使用者」持有这个内存 inode。`get()` +1，`put()` -1。
- **`di_nlink`**（磁盘硬链接计数）：有多少个**目录项**指向这个 inode（即文件有几个名字）。`ln` +1，`rm` -1。

**`get(ino)`（`:60`）**：先查内存缓存（线性表 64 格，`cache_lookup`）；命中则 `i_count++` 返回同一份；未命中则占一个空槽、从磁盘 `read_dinode` 读进来、`i_count=1`。——**保证同一个 inode 在内存里只有一份**，否则两个使用者各改各的会丢更新。

**`put(ip)`（`:83`）**：`i_count--`；
- 还有人用（`>0`）→ 直接返回。
- 没人用了（`==0`）且 `di_nlink==0`（最后一个名字也被删了）→ **此时才真正销毁**：`truncate` 放所有数据块 → `di_mode=0` 写回 → `sb_.ifree` 释放 inode 号 → 释放缓存槽。
- 没人用但 `di_nlink>0`（还有别的名字）→ 若 dirty 就写回，仅释放缓存槽，**inode 本身不删**。

> **这解释了 `rm` 的语义**：`rm` 只做 `unlink`（删目录项）+ `di_nlink--`，并不立刻删数据；只有当 `nlink` 归 0 **且**没人打开它（`i_count==0`）时，`put` 才回收。这正是 UNIX「删一个还开着的文件，空间要等关闭才释放」的行为。
>
> **已知局限（可在报告"取舍"里写）**：① `ialloc` 在 50 格空闲 inode 栈耗尽后**不会**回扫描磁盘补充（`super_block.cpp:155` 注释明说是 TODO），所以实际可用 inode 数受这个缓存栈限制。② 目录的 `di_nlink` 故意保持 1，不维护经典的「2 + 子目录数」（`unix_fs.cpp:392` 注释说明这是 cosmetic，`rmdir` 靠判空而非靠 nlink）。

### 4.5 目录与路径解析 namei（OS 概念 #4，`directory.cpp`）

**目录就是一个特殊文件**，内容是 `DiskDirEntry[]` 数组，每项 16B = `d_name[14]` + `d_ino`（`d_ino==0` 表示空槽）。所以「目录操作」复用了 inode 的读写：`read_dir_data`/`write_dir_data`（`:35`/`:60`）通过 `bmap`/`bmap_alloc` 读写目录这个文件的数据块。

- `init_dir`（`:155`）：新目录头两项写 `.`(自己) 和 `..`(父)。
- `link`/`unlink`（`:104`/`:135`）：在目录数组里找空槽插入 / 找到名字把 `d_ino` 置 0。删除是**逻辑删**（留空槽复用），不收缩。
- `lookup`（`:90`）：在一个目录 inode 里线性找名字 → inode 号。

**`namei(path, cwd, root)`（`:183`）—— 路径名解析，文件系统的「心脏」**：
```
绝对路径(/开头) 从 root 起；相对路径从 cwd 起。
按 / 切成 ["home","alice","a.txt"]，逐段：
  get(当前inode) → 确认是目录 → lookup(下一段) → 拿到下一段inode → put
任一段不存在/不是目录 → 返回 INVALID_BLK
```
`namei_parent`（`:208`）是变体：返回**父目录** inode + 最后一段名字（basename），给 create/mkdir/delete 用——它们需要在父目录里插/删条目。

> **注意**：`split_path` 只切 `/`，**不处理 `.` / `..`**。`..` 的真正回退是在更上层 `UnixFs::fs_chdir`（`unix_fs.cpp:454`）里对**路径字符串**做的栈式处理（遇 `..` 弹一段）；而 inode 层的 `..` 因为目录里真有 `..` 条目所以也能 `lookup` 到。两条路径维护：`cwd_ino_`（真实定位用）和 `cwd_path_`（显示用，`pwd`）。

### 4.6 读写流程串起来（`unix_fs.cpp`）
- `fs_create`（`:91`）：namei_parent 找父 → 权限检查 → 查重 → `imng_.alloc` 建 inode → `dmng_.link` 挂进父目录。
- `fs_open`（`:134`）：`namei_follow`（跟随符号链接）→ 权限检查 → `oft_.alloc_fd` 在打开文件表分配 fd；`O_APPEND` 时把偏移设到文件尾。
- `fs_read`/`fs_write`（`:176`/`:228`）：按字节循环，逐块 `bmap`/`bmap_alloc` → `dev_.read/write_block`，跨块自动处理块内偏移；写不满一块时先读出再改（read-modify-write）。
- `sync()`（`:821`）：设了 `disk_path_` 时，每个写操作后 flush 超级块 + 整盘 dump 到 img，**防崩溃丢数据**（代价是每次写都全盘落地，慢但稳）。

### 4.7 权限检查 `check_access`（`unix_fs.cpp:829`）
```
cur_uid==0 (root) → 直接放行（root 绕过一切检查）
否则按身份选三组权限位之一：
  我是 owner → 用 PERM_U* (rwx 高位)
  我同 gid   → 用 PERM_G*
  其它       → 用 PERM_O*
required 含 O_READ 必须有 r 位；含 O_WRITE/O_APPEND 必须有 w 位。
```
注意 `O_APPEND` 也要 w 位（`:849` 注释：否则追加会绕过权限）。`set_user(uid,gid)` 由 login/su 驱动，把当前身份灌进引擎。

### 4.8 链接的实现
- **硬链接 `fs_link`（`:619`）**：在目标目录加一条指向**同一个 inode** 的目录项，`di_nlink++`。两个名字共享数据。禁止给目录做硬链接。
- **符号链接 `fs_symlink`（`:660`）**：新建一个 `MODE_SYMLINK` 的 inode，**把目标路径字符串写进它的数据块**。解析时 `namei_follow`（`:756`）读出字符串再 `namei` 一次，带 16 跳上限防环。`stat`/`ls` 用 lstat 语义（报告链接本身而非目标），`open`/`cd` 才跟随。

## 5. FAT16 引擎 —— 对照学习（B 写，`fat16_fs.cpp`）

价值在于：**同一个 IFileSystem 接口，两种完全不同的实现**。对照能让你真正理解 VFS 抽象。

### 5.1 布局与结构
```
块 0      1..4     5..8     9          10 ........... 545
┌──────┬────────┬────────┬──────────┬──────────────────┐
│boot  │ FAT1   │ FAT2   │根目录(1块)│ 数据区(簇=块,1:1) │
│      │ 4 块    │ 备份    │16个目录项 │ 簇号 2..537       │
└──────┴────────┴────────┴──────────┴──────────────────┘
簇→块：cluster_to_block(c) = 10 + (c-2)
```
- **FAT 表**：`uint16_t fat_[]`，每项对应一个簇。`0x0000`=空闲，`0xFFFF`=链尾(EOC)，`0xFFF7`=坏簇，2..N=指向下一簇。两份 FAT 互为备份，`flush_fat` 同步写。
- **目录项 32B**（`FAT16DirEntry`）：`name[8]+ext[3]`（8.3 大写）、attr、time/date（DOS 编码）、`first_cluster`、`file_size`。删除用 `name[0]=0xE5` 标记。

### 5.2 簇链分配（对照成组链）
- `alloc_cluster`：从簇 2 **线性扫描**找第一个 `0x0000`，置 EOC 返回。**O(n)**，无栈无位图。
- `free_cluster_chain`：顺着链把每个簇置 0。
- `fs_write` 增量 I/O：定位到 offset 所在簇（不够就分配新簇接到链尾），read-modify-write，必要时延长链，最后更新目录项里的 `file_size`。

### 5.3 与 UNIX 引擎的本质差异（必答题 Q3）

| 维度 | UNIX inode | FAT16 |
|------|-----------|-------|
| 索引方式 | inode 混合索引（直接+间址） | FAT 表链表 |
| 空闲管理 | 成组链 + 空闲 inode 栈 | 线性扫描 FAT |
| 元数据位置 | 集中在 inode 区 | 分散在目录项 + FAT |
| 文件名 | 14 字符，名与 inode 分离 | 8.3 大写，名与元数据同在目录项 |
| 权限 | 完整 rwx（owner/group/other） | 无（`fs_chmod` 是空操作返回 0） |
| 硬/软链接 | 支持 | **不支持**（`fs_link/symlink` 返回 -1） |
| 随机访问大文件 | 间址，O(1)~O(级数) | 必须顺链，O(n) |

> **答辩点**：FAT 简单、对小盘/可移动介质友好，但随机访问大文件慢（要顺着簇链走）、无原生权限；UNIX inode 随机访问快、元数据丰富，但结构复杂。这就是为什么课设让你做两个——对比才显功力。

## 6. VFS 抽象层 + 打开文件表

### 6.1 IFileSystem 多态如何让引擎可切换（`vfs.h`）
`IFileSystem` 是纯虚接口（`fs_format/open/read/write/mkdir/...` 全 `=0`）。`UnixFs` 和 `Fat16Fs` 各自 `override`。上层只持有 `IFileSystem*`，**编译期不知道、运行期才决定**是哪种引擎——TUI 按 F2 时就是把这个指针从一个换成另一个（`switch_fs`）。这就是「面向接口编程」「依赖倒置」在 C++ 里的落地。

接口里几个为 TUI/权限服务的扩展方法值得注意：`set_user`（登录态注入）、`fs_block_map`（给 F3 磁盘位图填每块状态 FREE/USED/META）、`fs_disk_usage`（用量条）、`fs_symlink/fs_readlink`（FAT16 直接返回 -1）。

### 6.2 系统/用户打开文件表（OS 经典双层结构，`open_file_table.cpp`）
```
用户打开文件表(每用户)            系统打开文件表(全局40项)
ofile[fd] = sys_idx  ──────────▶ SysOpenFileEntry{
  (-1=空)                           f_flag  打开模式
                                    f_count 引用计数(可被多 fd/多用户共享)
                                    f_inode_no
                                    f_offset 读写指针 }
```
- `alloc_fd`：在用户表找空 fd → 在系统表找 `f_count==0` 的项 → 关联。
- 读写偏移 `f_offset` 存在**系统表**，所以 dup/共享时偏移是共享的（符合 UNIX 语义）。
- `close_all`：logout 时把该用户所有 fd 释放。

> **答辩点**：为什么两层？用户表给每个进程/用户私有的 fd 空间（小整数 0,1,2...）；系统表让多个 fd 能共享同一个打开文件的偏移和模式。这是 UNIX 文件描述符机制的教科书结构。

## 7. Shell 层（`command_registry.cpp` + `main.cpp`）

- **命令表**：`register_cmd(name, handler, usage)` 把命令存进有序 vector。handler 签名统一为
  `int(IFileSystem& fs, UserManager& um, vector<string> args, string& output)`，返回 0/-1，输出写进 `output`（TUI/CLI 负责显示）。
- **`execute`**：分词 → glob 展开 → 查表 → 自动支持 `-h/--help`（打印 usage）→ `record` 记日志 → 调 handler → 捕获 `stoi` 异常转成错误串（防坏参数崩溃）。
- **分词 `tokenize_ex`**：空白切分；双引号内不展开 glob，支持 `\n \t \\ \"` 转义。
- **glob `expand_globs` + `glob_match`**：经典 `*`/`?` 回溯匹配，**只对路径最后一段**展开（`docs/*.log` 在 docs 里展开，`a*/b` 不展开）；无匹配则原样返回（nullglob off）。
- **日志/回放**：`record` 把 `时间戳\t用户\t命令行` 追加到日志文件（跳过 log/replay/help/history/exit 这些元命令）；`replay` 快照后逐条重放，期间 `suppress_log_` 防重复记日志。
- **全部 ≈32 条命令**：用户(login/logout/useradd/passwd/su)、目录(mkdir/rmdir/cd/pwd/ls/ll/tree)、文件(touch/rm/cp/mv/cat/more/find)、I/O(open/close/read/write)、元/权限(chmod/ln/readlink/stat/disk/format)、系统(help/log/replay/history/exit)。
- **main.cpp**：解析 `--tui/--fat16/--format` → 建对象 → 注册所有命令 → 进 CLI 主循环或 `tui.run()`。**CLI 用所选引擎 + 对应 img（pfs_unix.img / pfs_fat16.img）**；`more` 命令通过给输出加 `\x01PFS_PAGE\x01` 前缀触发分页 `cli_page`。

## 8. TUI 层 —— ncurses 速成（B 写，你不熟，重点讲，`tui.cpp`）

### 8.1 ncurses 是什么
一个操控终端「字符画布」的库：你不再 `printf` 到行尾滚动，而是**在一个二维网格上任意 (y,x) 定位、画框、上色、分窗口**，最后统一刷新。核心心智：**「窗口对象 + 改缓冲 + refresh 才上屏」**。

### 8.2 面板布局（六个窗口，必画图 #3 之一）
```
┌─ 标题栏(1行): PFS ─ 引擎 ─ user ─ 时钟 ───────────┐
├──────────────────────┬───────────────────────────┤
│ 目录树 tree_win       │ 磁盘用量 disk_win          │  上半 55%
│ (递归画 ├─└─│)        │ (块/inode 进度条 draw_meter)│
├──────────────────────┴───────────────────────────┤
│ 终端框 term_box ┌ 内含可滚动 term_win(derwin) ──┐  │  下半
│                 │ 命令输入/输出在这里            │  │
├─────────────────────────────────────────────────┤
│ 状态栏(1行): F1帮助 F2切FS F3磁盘 F5刷新 F10退出  │
└─────────────────────────────────────────────────┘
```

### 8.3 必会的 ncurses API（对照 tui.cpp 用法）
| API | 作用 |
|-----|------|
| `initscr / endwin` | 进入/退出 ncurses 模式 |
| `cbreak / noecho` | 不等回车即拿键 / 不自动回显（自己控制） |
| `keypad(win,TRUE)` | 让 F1/方向键返回 `KEY_F(1)`/`KEY_UP` 而非转义序列 |
| `newwin(h,w,y,x)` | 建窗口；`derwin(parent,...)` 建共享父缓冲的子窗 |
| `box(win,0,0)` | 画边框 |
| `mvwprintw / waddch / waddstr` | 定位输出 |
| `wattron/off + COLOR_PAIR(n)` | 开关属性/颜色（项目初始化了绿/蓝/黄/青/红 5 对） |
| `wrefresh / touchwin / werase` | 刷新上屏 / 强制重绘 / 清窗 |
| `wgetch(win)` | 读一个键；`wtimeout(win,1000)` 让它每秒超时一次（驱动时钟刷新） |
| `scrollok(win,TRUE)` | 允许终端窗滚动 |

### 8.4 事件循环骨架（`run`，`:402`）
```
initscr → 配色/keypad → 建六个窗 → 画初始面板
loop:
  ch = wgetch(term_win)           // 每秒超时刷新时钟
  switch(ch):
    回车   → 执行命令 reg_->execute(...) → 刷新 tree/disk 面板
    退格   → input_buf 删尾
    ↑/↓    → 命令历史
    F1     → 帮助弹窗(居中 newwin, keypad 防 ESC 泄漏)
    F2     → switch_fs(): 卸当前→挂另一引擎→换指针(各自独立磁盘镜像!)
    F3     → fs_block_map() 拿每块状态 → 全屏画位图(红=用,青=元,绿=空)
    F5     → redraw_all()
    F10    → running_=false 退出
    KEY_RESIZE → 删窗重建
    可打印 → 入 input_buf 并回显
endwin
```
> **F2 切换为何安全**：UNIX 的 inode 布局和 FAT16 的 FAT 布局互不兼容，共用一块「磁盘」会互相踩坏。所以 **TUI 给每个引擎各一个独立 `BlockDevice` + 独立镜像**（`pfs_tui_unix.img` / `pfs_tui_fat16.img`），切换只是换指针，互不干扰。

## 11. 验收 / 答辩准备（冲刺优秀）

### 11.1 四个必答问题（标准答案骨架）
1. **文件卷的组织结构** → 画 §4.1（UNIX）和 §5.1（FAT16）两张布局图，口述每个区干嘛。
2. **主要数据结构间的关系** → `SuperBlock`(空闲管理) ↔ `inode`(索引+元数据) ↔ `DirEntry`(名字→inode号) ↔ 数据块；FAT 版：目录项(首簇) → FAT 表(簇链) → 数据簇。
3. **格式化做了什么** → ① 清零盘 ② 建超级块、建**成组空闲块链** ③ 建空闲 inode 栈 ④ 建根目录(写 `.`/`..`) ⑤ 预置 /etc /home（见 `fs_format` `unix_fs.cpp:25`）。
4. **哪些因素决定可管理的最大文件长度** → 块大小(512)、`di_addr` 数组大小(10)、间址级数(直接/一次/二次)、间址块容量(256 指针)、数据区总块数(512)。算出 §1.4 的 3KB+256KB+32MB（但受 512 数据块物理上限制约）。

### 11.2 你必须能徒手画的三张图
1. UNIX 磁盘布局（boot/super/inode/data 块号）
2. 成组链一次 `balloc`：栈剩 1 项时读磁盘组、返回链块本身
3. 混合索引寻址树：addr[0..5] 直接 / [6][7] 一次 / [8] 二次

### 11.3 可能被追问的刁钻问题（带答案）
- **「`rm` 一个还 open 着的文件会怎样？」** → 只 `unlink` + `nlink--`；因为 `i_count>0`，`put` 不回收，数据块还在；等 `close` 使 `i_count==0` 且 `nlink==0` 才真正 `truncate`+`ifree`。（§4.4）
- **「`di_addr[i]==0` 不会和真实块号 0 冲突吗？」** → 不会。format 预留了数据区前 3 块，`balloc` 永不发出相对块号 0，所以 0 专用于表示空洞。（§4.1）
- **「删大文件如果只放直接块会怎样？」** → 间址块和它指向的数据块全泄漏；`truncate` 必须逐级遍历释放。（§4.3）
- **「成组链能 O(1) 判断第 100 块是否空闲吗？」** → 不能，要遍历整条链（`collect_free_blocks`）。这是相对位图的劣势。（§4.2）
- **「FAT16 为什么没有 chmod 效果？」** → FAT 目录项没有 rwx 位，`fs_chmod` 故意返回 0（空操作）保持 shell 行为统一。（§5.3）
- **「两个引擎共用一块磁盘行吗？」** → CLI 下一次只用一种引擎、各自镜像；TUI 要切换，所以给每引擎独立 `BlockDevice`+镜像，否则布局互相破坏。（§8.4）
- **「inode 缓存满了 / 空闲 inode 栈空了会怎样？」** → 缓存满 `cache_alloc` 返回 null；空闲栈空 `ialloc` 返回 `INVALID_BLK` 且当前**不回扫磁盘补充**（已知 TODO）。（§4.4）

### 11.4 已知取舍 / 局限清单（报告「设计权衡」段可直接用）
- 空闲 inode 栈耗尽不回扫磁盘（`super_block.cpp:155`）。
- 目录 `nlink` 恒为 1，不维护经典计数（`unix_fs.cpp:392`）。
- `sync()` 每次写全盘落地——稳但慢。
- 密码明文存储。
- FAT16 不支持链接、无权限、根目录固定 16 项、簇分配线性扫描 O(n)、文件名大写 8.3。
- 目录项删除是逻辑删（留空槽），不收缩目录文件。

### 11.5 参考源码与本项目的关系（`协作手册.md §5`）
- `../filesysorig/`：老师给的 C 样例，数据结构/函数名与 PPT 对应，但**有 7 处已知 bug**（namei 返回 0 与合法索引 0 冲突、bfree 缺参、iput 误用 balloc、iname 参数方向、_dir 变量冲突、open 判断取反、login strcmp 判断错）。本项目相当于「修正并 C++ 化」了它——**这些 bug 修复点是报告里的亮点**。
