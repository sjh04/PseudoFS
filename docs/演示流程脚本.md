# PseudoFS 现场演示脚本(主操作手:A)

> 全部命令已用 `build/pfs` **实测跑通**,下方「→」后是**真实输出**,照着念即可。
> 设计目标:10~12 分钟覆盖**全部 35 条命令 + 所有评分点**(目录 / 文件 I/O / 打开文件表 / 权限 / 链接 / cp-mv / 通配符 / 查找 / **大文件三级混合索引** / 多用户访问控制 / **日志回放** / 持久化 / 双引擎+TUI)。
> 每个 Act 末标注它**证明了哪个功能点 / 必答题**。文末有「命令覆盖对照表」核对不漏命令。

## 0. 开演前检查(60 秒,务必先做)

```bash
cd PseudoFS/build && cmake .. && make -j$(nproc)   # 确认能编出 pfs
ls pfs                                              # 可执行在
```
- 终端拉大到至少 **100×30**(TUI 四面板才不挤)。
- 干净开场:演示用 `./pfs --format`(强制格式化,从全 0 裸盘开始)。
- **大文件那一幕(Act 7)需要提前 prep**——见该节,30 秒生成好 `big.txt`。
- **持久化那一幕(Act 10)反过来——不要加 `--format`**,否则盘被重置就演不出"重启数据还在"。

> ⚠️ 四个最易翻车点,先记牢:
> 1. **`open` 返回的 fd 是 `0`,不是 3**。后面 `write`/`close` 跟着屏幕上的真实数字念。
> 2. **写完别用同一个 fd 去 `read`**——偏移已在文件末尾,读不到内容。要么 `close` 后 `cat`,要么重新 `open ... r` 再 `read`。
> 3. **`open` 不会自动建文件**——必须先 `touch`,再 `open`。
> 4. **Act 8 的 `permission denied` 是预期结果,不是 bug**——要主动说"看,权限系统把它拦住了"。

## Act 2 — 文件 I/O + 打开文件表

```
touch notes.txt
open notes.txt rw
write 0 Hello PseudoFS -- OS course demo
close 0
cat notes.txt
```
→ `open` → **`fd=0`**(念这个数字!)
→ `write 0 ...` → `32 bytes written`
→ `cat` → `Hello PseudoFS -- OS course demo`

**话术**:「`touch` 建空文件;`open` 在打开文件表分配一个 fd(这里是 0),`write` 通过它定位 inode、按混合索引把数据落到数据块,`close` 释放 fd。」
**(可选)纯读路径**:
```
open notes.txt r
read 0 100
close 0
```
→ `read 0 100` → `Hello PseudoFS -- OS course demo`
**证明**:E-01 touch · E-02 open · E-03 read · E-04 write · E-05 close · 打开文件表。

## Act 3 — 权限(rwx 访问控制)

```
ll
chmod 600 notes.txt
ll
```
→ 第一次 `ll`:`-rwxrwxrwx  1 root  32  ...  notes.txt`
→ `chmod 600` 后:`-rw-------  1 root  32  ...  notes.txt`

**话术**:「权限位实打实存在 inode 的 mode 里;`chmod 600` 后 owner 只读写、group/other 全无——Act 8 就用它演真正的访问拦截。」
**证明**:A-05 文件访问控制 · C-01 chmod · D-05 ll。

## Act 4 — 链接(UNIX 引擎亮点,FAT16 做不到)

```
ln notes.txt hard.txt
stat notes.txt
ln -s notes.txt soft.txt
readlink soft.txt
ls
```
→ `stat notes.txt` → `Links: 2`(硬链接让 nlink 变 2)
→ `readlink soft.txt` → `notes.txt`
→ `ls` → `./  ../  notes.txt  hard.txt  soft.txt@`(`@` 标记软链接)

**话术**:「硬链接 `hard.txt` 和 `notes.txt` **指向同一个 inode**,所以 nlink=2、共享数据;软链接 `soft.txt` 是另一个 inode,里面存的是目标路径字符串。这正是 UNIX 引擎比 FAT16 强的地方——FAT16 不支持链接。」
**证明**:A-04 硬/软链接 · B-03 stat · D-04 ls。

## Act 5 — 复制 / 移动 / 通配符 / 查找 / 磁盘用量

```
cp notes.txt copy.txt
mv copy.txt renamed.txt
ls
ls *.txt
find / notes.txt
disk
```
→ `cp` → `Copied.` · `mv` → `Moved.`
→ `ls` → `./  ../  notes.txt  hard.txt  soft.txt@  renamed.txt`
→ `ls *.txt` → `hard.txt  notes.txt  renamed.txt  soft.txt@`(通配符只展开 `.txt`)
→ `find / notes.txt` → `/home/alice/notes.txt`
→ `disk` → `FS: UNIX` / `Blocks: 10/2048 used` / `Inodes: 8/256 used`

**话术**:「`ls *.txt` 是 shell 风格通配符展开;`find` 从指定目录递归按名搜索;`disk` 实时显示成组链管理下的块用量和 inode 用量。注意数据区是 **2048 块**——这是为了 Act 7 能演到二次间址特意扩的。」
**证明**:A-03 cp · B-01 mv · C-02 通配符 · B-06 find · B-04 磁盘可视化(数据侧)。

## Act 6 — 删除(空目录 / 文件 / 递归)

```
more notes.txt
mkdir tmp
rmdir tmp
rm hard.txt
ls
```
→ `more notes.txt` → 分页显示内容
→ `rmdir tmp` → 成功(空目录,无输出)
→ `rm hard.txt` → 成功(无输出)
→ `ls` → `./  ../  notes.txt  soft.txt@  renamed.txt`(hard.txt 没了)

**话术**:「`more` 分页查看;`rmdir` 只删空目录(非空会被拒);`rm` 删文件,`rm -r` 可递归删整棵目录树。删 `hard.txt` 只是去掉一个目录项、把 nlink 减回 1,`notes.txt` 的数据还在——这就是硬链接的意义。」
**(可选)演递归删**:`mkdir -p a/b/c ; rm -r a ; ls`(整棵子树消失)
**证明**:B-02 more · D-02 rmdir · E-06 rm / rm -r。

## Act 7 — 大文件 + 混合索引三级寻址(A-02 加分核心,必答题 Q4)★

> **这一幕是和参考项目拉开差距的关键**:直接索引 → 一次间址 → 二次间址三级全演。
> 300KB 文件没法现场敲,**演示前 30 秒** prep 好(单独开一盘):

**Prep(开演前在另一个终端跑一次):**
```bash
cd PseudoFS/build
python3 -c "
print('touch /big.txt'); print('open /big.txt w')
for _ in range(300): print('write 0 '+'B'*1000)
print('close 0'); print('exit')
" | ./pfs --format
```
这写出一个 **300000 字节(≈293KB)** 的 `/big.txt` 并存进 `pfs_unix.img`。

**现场(`./pfs` 不加 --format,直接读上面那盘):**
```
stat /big.txt
disk
open /big.txt r
read 0 40
close 0
```
→ `stat` → `Size: 300000 bytes`
→ `disk` → `Blocks: 596/2048 used` · `Inodes: 5/256 used`
→ `read 0 40` → `BBBBBBBB...`(40 个 B,证明读回无损)

**铁证——dump inode 的地址数组**(big.txt = inode 4):
```bash
python3 -c "
import struct
d=open('pfs_unix.img','rb').read(); di=d[2*512+256:2*512+256+64]
a=struct.unpack_from('<10H',di,24)
print('addr[0..5] 直接   :',a[0:6])
print('addr[6]    一次间址:',a[6])
print('addr[7]    一次间址:',a[7])
print('addr[8]    二次间址:',a[8],'  <== 已启用!')
"
```
→
```
addr[0..5] 直接   : (2044, 2043, 2042, 2041, 2040, 2039)
addr[6]    一次间址: 2038
addr[7]    一次间址: 1782
addr[8]    二次间址: 1525   <== 已启用!
```
（也可用 `/disk-debug inode 4` 看同一份数据。)

**话术**:「一个 inode 的地址数组 `addr[10]`:前 6 项直接指数据块(≤3KB);`addr[6]`、`addr[7]` 各指一个一次间址块,每块再装 256 个块号(各覆盖 128KB);`addr[8]` 是二次间址,指向一个'指针块的指针块'。这个 293KB 的文件 **三级全用上了**——你看 `addr[8]` 非零,说明它跨过了第 518 个数据块,真正走进了二次间址。
块账也对得上:586 个数据块 + 4 个间址块(两个一次间址 + 一个二次间址根 + 一个二级块)+ 初始 6 块 = 596。
顺便:`addr` 里的块号都是高号(2044、2043…),因为成组链接法的 balloc 先发内存栈里的高号块——这是它和位图顺序分配的行为差异。」

> 答辩 Q4「哪些因素影响文件最大长度」:块大小(512B)、地址数组项数(10)、间址级数(直接/一次/二次)、数据区总块数(2048)——四个全在这一幕里看得见。

**证明**:A-02 混合索引(直接+一次+二次间址)· 必答题 Q4 · 大文件支持。

## Act 8 — 多用户 + 访问控制拦截 + 改密/登录(全场高光)

```
useradd alice alice 1000 1000
su alice alice
cat /home/alice/notes.txt
su root root
```
→ `useradd` → `User alice created.`
→ `su alice alice` → `Switched to alice`
→ `cat ...` → **`cat: /home/alice/notes.txt: permission denied`**
→ `su root root` → `Switched to root`

**话术**:「`notes.txt` 是 root 的、权限 600。切到 alice 再读它——**被拒绝**。权限检查真的在 inode 层生效:alice 不是 owner、gid 也不同,落到 other 位、无 r 权限。切回 root,root 绕过一切检查,又能读了。」

**改密码 + 登录态演示**:
```
passwd root newroot
logout
login root newroot
```
→ `passwd` → `Password changed.`
→ `logout` → `Logged out.`
→ `login root newroot` → `Welcome, root (uid=0)`

**话术**:「`passwd <旧> <新>` 改密码并自动落盘;`logout` 丢掉会话,`login` 重新验证密码进入。账户表独立持久化在 `users.dat`。」
**证明**:U-01 login · U-02 logout · U-03 useradd · U-04 passwd · B-05 su · A-05 权限检查(真实拦截 + root 绕过)。

## Act 9 — 操作日志与回放(C-05)★

```
history
log
```
→ `history` → 带行号的命令历史
→ `log` → 带时间戳和用户名的操作日志(`mkdir`/`touch`/… 真正改动文件系统的操作)

**回放的杀手锏——日志跨会话持久,能在格式化后的空盘上重建一切:**

> 在 Act 10 退出后、或单独演:`./pfs --format`(磁盘被抹掉,但 `pfs_ops.log` 还在)
```
ls /
replay
tree /proj
```
→ `ls /` → 只剩 `etc/  home/`(之前建的都没了)
→ `replay` → `Replayed N operation(s).`
→ `tree /proj` → 结构**原样重建**

**话术**:「每条改动文件系统的命令都记进 `pfs_ops.log`,跨会话累积。`replay` 把它们按序重放——哪怕磁盘刚被格式化成空盘,也能把整个目录结构重建出来。这就是操作日志 + 回放的价值。」
**证明**:C-03 命令历史(history,↑↓ 翻阅)· C-05 操作日志 + 回放。

## Act 11 — 双引擎 + TUI(B 协同;最有视觉冲击,放压轴)

```
exit
./pfs --tui
```
TUI 里依次按:
- 看四面板:**目录树** / **磁盘用量条** / **终端** / 标题栏时钟。
- **F2** → 切到 FAT16 引擎(标题栏 `[UNIX]` 变 `[FAT16]`)。FAT16 下 `mkdir`/`touch`/`write`/`cat` 同样跑通——**同一套命令、两种底层实现**(VFS 多态)。
- **F3** → 全屏磁盘块位图(红=已用 / 青=元数据 / 绿=空闲),数据区 2048 块按屏宽采样显示。
- **F5** 刷新 · **F10** 退出。

**话术**:「F2 把底层从 inode 体系换成 FAT 表,上层命令一行没改——靠的是 `IFileSystem` 抽象接口。两个引擎各有独立磁盘镜像,互不踩坏。FAT 表也随盘增大扩到了 9 块,覆盖全部数据簇。」
**证明**:A-01 双引擎 · A-06 TUI · B-04 磁盘可视化 · VFS 多态。

> TUI 由 B 写,这一幕可由 **B 操作、你解说**。F 键若被 SSH/终端截获,用菜单提示的替代键,或退回 CLI `./pfs --fat16` 跑 FAT16 简版。
